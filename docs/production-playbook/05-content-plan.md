# 05. 内容产出战略（核心产出）

> **定位**：本文档是本专题的**第一公民**。其他文档（01 原则 / 02 ROI / 03 部署 / 04 陈述精度）都是为内容产出服务的。
> **逻辑链**：真实运维 → 积累素材 → 持续输出 → 建立专业信用 → 作品集 / 潜在机会都是自然结果。

---

## 为什么内容产出是第一公民

在 AI 时代，工程师的**长期复利资产**只有两类：

1. **真实运维时间**（不可压缩，靠 Phase 5 积累）
2. **持续公开输出**（复利，基于 1 产生）

简历、面试、接项目——都是**从这两条派生的副产出**。方向反过来就会走偏：
- 把简历作为北极星 → 写成包装话术 → 深挖穿帮
- 把内容产出作为北极星 → 倒推需要真实经验 → 自然导向真运维 → 最终副产出全部到位

→ **先做内容生产者，再做求职候选人**。顺序不能反。

## 输出原则

### 1. 内容基于真实经验，拒绝"教程翻译"

**❌ 不写**：
- "Langfuse 快速入门"（文档翻译，AI 10 秒产出同等）
- "Docker 最佳实践"（AI 能写 100 篇）
- "OpenTelemetry 原理"（教科书重复）

**✅ 要写**：
- "Langfuse 自托管 v3 踩坑实录：我花了 3 天才发现的 5 个问题"
- "双节点跨云部署 AI Agent 的决策过程"
- "trace_id 三端贯穿的真实实现与调试"

**判断标准（写之前自问）**：
- 这篇 AI 能代写 80% 吗？能 → 不写，或者加入独家经验直到它代写不了
- 这篇需要我有什么独家资产？（commit 链接？日志截图？事故时间线？）
- 有独家资产 → 保留
- 没有 → 去积累，或者换选题

### 2. 单篇质量 >> 发布频率

不追求：
- 10 万+ 阅读
- 粉丝数 / 社交互动
- 日更 / 周更打卡

追求：
- 单篇能支撑 15 分钟深度讨论
- 2 年后回看这篇依然不过时
- 读过的人愿意主动分享 / 引用

**单篇精修 1 个月 > 月更 4 篇水文**。

### 3. 内容服务于专业信用，不服务于流量

**专业信用 = 读者记住你在某领域靠谱**。

- 读过你 3 篇 Langfuse 相关深度文章 → 他下次遇到 Langfuse 问题会想到你
- 读过你 3 篇 ROI 决策方法论 → 他会推荐你给做选型的朋友

这种信用是 4-6 个月才能攒起来的——选题**集中领域 + 持续深化**，不要跳题。

## 内容类型与价值矩阵

| 类型 | 描述 | 价值 | 产出成本 | 依赖 |
|------|------|------|---------|-----|
| **A. 架构决策复盘** | "我为什么这么做 + 替代方案为什么不选" | **高** | 中 | 本专题 docs 现成素材 |
| **B. 踩坑实录** | 现象 → 排查 → 根因 → 修复 → 教训 | **最高** | 低 | Phase 5 真实事故 |
| **C. 技术深度解读** | 从源码/spec 出发讲某机制 | 中高 | 高 | 独家实操经验 |
| **D. Demo Video** | 10-15 分钟讲清一个技术点 | 高 | 最高 | Phase 6 集中录制 |
| **E. L6 地图公开版** | 某领域的认知地图 | 中 | 低 | `knowledge-maps/` 现成 |
| **F. 方法论类** | ROI 框架 / L1-L6 分级 / 调研纪律等 | 中高 | 低 | 本专题 docs 现成 |

**建议分布**（总共 5-8 篇 + 2-3 video）：
- A 类：2-3 篇（现在就能写）
- B 类：2-3 篇（Phase 5 后产出）
- C 类：1-2 篇
- D 类：2-3 个 video
- E 类：2-3 篇（配合知识地图填充后发）
- F 类：1-2 篇（最独特，容易被记住）

## 叙事模板：同一事故 × 三种输出形式

一个真实事故可以同时产出 blog / 演讲 / demo video。不要让一份素材只用一次。

**案例：ClickHouse 磁盘爆炸事故（假设 Phase 5 遇到）**

### Blog 版（长文，2000-3000 字）
- 前置：我的 Langfuse 生产架构 + ClickHouse 角色
- 现象：Telegram 告警 + Grafana 截图（磁盘 97%）
- 排查时间线：每一步命令 + 输出 + 当时的判断
- 根因：TTL 配置的具体陷阱（引用 ClickHouse 文档 + 我读源码的发现）
- 修复：临时 DROP PARTITION + 永久 ALTER TABLE + 改进 TTL 模板
- 教训：5 点抽象，不只是"加告警"这种肤浅总结
- 附录：如何自己复现 + 容量规划 excel 模板

### 演讲版（15-20 分钟 slide）
- 3 分钟：场景 + 现象（带截图冲击）
- 5 分钟：关键排查节点 + 我当时怎么想
- 3 分钟：根因（就讲 1 个核心陷阱，不贪多）
- 2 分钟：修复
- 5 分钟：更通用的教训 + 给听众的 checklist（演讲最值钱的部分）
- 2 分钟：Q&A hook

### Demo Video 版（8-10 分钟）
- 开头 30 秒：结果先行——"凌晨这条 alert 我 42 分钟恢复，流程是..."
- 主体：录屏重演一遍（快进）
- 结尾 1 分钟：抽象教训 + 讲稿文件链接

**一份 40 小时的事故积累 → 3 种形式 → 3 波读者触达**。

## 发布渠道矩阵

| 渠道 | 特点 | 建议投入 |
|------|------|---------|
| **个人博客**（自建 / Hashnode / Vercel） | 主阵地，沉淀长文 + SEO | **重点** |

| **知乎** | SEO 价值 | 低（可选） |

1. 个人博客首发（有时间戳 + 首发证明）
2. 掘金同步（手动搬运）
3. X/Twitter 发简介 + 链接
4. 知乎/Medium 视需要

## 质量自检 checklist

每篇发布前过一遍：

- [ ] **原创性**：AI 能代写 80% 以上吗？如果能，加入独特经验或删掉
- [ ] **数据**：有具体数字吗（时间 / 量级 / 命令 / 版本号 / 百分比）？
- [ ] **可验证**：有 GitHub commit / 日志截图 / 代码 snippet？
- [ ] **完整性**：读者读完能复现你的结论吗？（不是"读完觉得你厉害"）
- [ ] **独立意义**：脱离本项目，这篇对读者还有独立价值吗？
- [ ] **读者追问测试**：如果被留言追问 3 轮我能答吗？
- [ ] **可引用性**：2 年后有读者搜到这篇，依然可用吗？

## 写作规律建议

### 触发式写作（推荐）

- 遇到真实事故 → 48 小时内先写 draft（记忆最清楚时）
- 先粗糙 outline + 关键命令 + 时间线
- 等 2-3 天冷静下来再精修
- 一个月总结时合并多个 draft

### 批量写作（避免）

- 不要"周末坐下来憋 3 篇"——闭门造车容易流于教程式
- 不要"为了凑数"——宁缺毋滥

### 段落性交付（推荐）

- 每篇 blog 不一次写完
- 第一次：outline + 最难写的 1-2 段
- 第二次：填充其他段
- 第三次：精修 + 配图 + 互相链接
