# 01. 核心原则：AI 时代的职业护城河

> **一句话**：工程师的可分享价值 = 这段经验对 AI 的替代难度。

---

## 为什么需要这个原则

在 AI 辅助编程大规模普及后，"会用某工具"、"搭过某栈"这类经验正在快速贬值。

原因：

- Claude / Cursor / Copilot 可以在几小时内带一个完全不懂的人本地搭起一套 Langfuse / LiteLLM / OTel
- 读者（包括懂 AI 的同行、技术评估者）默认候选人的"本地搭建经历"有 70% 是 AI 完成的
- 自述"熟悉 X 技术"已经不能证明任何事——读者会直接让 AI 生成同样内容做对照

→ **评估一段经验值不值得投入的新标准不是"难不难"，而是"AI 替代不了吗"**。

---

## 技能按 AI 替代难度的 6 级分类

前 5 级是**单技术点的纵向深度**，L6 是**跨技术的元认知地图能力**，维度不同。

| 级别   | 描述                               | 示例                                                | 分享/陈述价值 |
| ------ | ---------------------------------- | --------------------------------------------------- | -------- |
| **L1** | AI 一次对话完成                    | "用 Next.js 写个 CRUD"                              | ~0       |
| **L2** | AI 能带你走完全流程                | "本地 docker-compose 起一套 Langfuse"               | 低       |
| **L3** | AI 能给方案但你要判断取舍          | "设计一个观测架构"                                  | 中       |
| **L4** | AI 能出主意但执行要真实环境 + 时间 | "生产运维 4 个月"                                   | **高**   |
| **L5** | AI 完全无法替代                    | "凌晨 3 点挂了怎么 15 分钟恢复"、"一次真实事故复盘" | **极高** |
| **L6** | **元认知层级**：知道自己在每个领域里**知道什么 / 不知道什么 / 以及承认必然有不知道自己不知道的** | "Langfuse 我运维过，LangSmith 读过文档没实操，Helicone 只听过没对比过，这领域还有我不知道的玩家" | **资深工程师级稀缺项** |

**L4-L5 的共同特征**（纵向深度）：

- **决策链路长**——一个问题要结合日志、监控、业务、架构多个维度判断
- **不可重放**——线上环境不能 reset，每次决策有真实代价
- **时间不可压缩**——ClickHouse 的磁盘满掉只有真运行 2 个月才会遇到

**L6 的共同特征**（横向地图 + 元认知）：

- **地图感**——对某领域能列出"我做过 / 读过 / 听过 / 没接触"四档分布
- **边界诚实**——用词精准到具体动作（"摸过"/"读过"/"听过"）不用"熟悉"
- **承认盲区**——知道地图上必然还有**没命名的未知区域**
- **决策留弹性**——做技术选型时主动留口子给"我不知道的方案"

## 自评陷阱：AI 辅助跑通 ≠ 技能掌握

**所有"AI 协作下跑通过流程"的经历，自评上限都是 L1**。

- 跟 Claude/Cursor 一起把 tts-agent-harness 部到 Fly.io 上线 → **L1**（不是"会 Fly.io"）
- 跟 AI 一起写 Dockerfile + docker-compose 跑起来 → **L1**（不是"会 Docker"）
- 跟 AI 一起配过 Cloudflare Pages + D1 → **L1**（不是"掌握 Cloudflare 生态"）

**为什么这些都只是 L1**：

- 没独立遇到过非文档化的问题
- 没在 AI 不在场时被逼着判断过
- 没经历过"改一个配置爆炸半天才修好"
- 没有可验证的时间积累和事故履历

**自评红线**（强制纪律）：

| 现状 | 能写的最高级别 |
|------|--------------|
| AI 辅助跑通过 + 没再动 | L1 |
| AI 辅助跑通 + 本地自己改配置玩过 | L2 |
| 线上跑 + 没遇过事故 | L2 |
| 线上跑 + 遇过 1-2 次真实问题独立定位修复 | L3 |
| 线上运维 3 月+ + 多次事故 + 有复盘 | **L4** |
| 上面 + 能讲清楚失败案例 + 源码级理解 + upstream 交互 | **L5** |
| 在某领域能画出自己的知识地图（做过/读过/听过/未接触）+ 能列出主流替代方案的了解程度 + 承认存在未知边界 | **L6**（元认知，和 L1-L5 正交） |

**任何对外陈述前过一遍这个表**。宁可写低不要虚高——虚高在深度追问里会穿帮，当场失去信用。

**常见自我欺骗**：

- "我 2 年前搭过" → 如果没持续用，已经退化到 L0
- "我跟着教程跑通过" → 教程规避了所有真实问题，停在 L1
- "我读过源码" → 没对应的实操使用，只算 L2 的知识面，不是 L3 的判断力
- "我用 AI 快速做了个 demo" → demo 不是生产，最多 L1

**这条原则和"学习 ROI 分级"正交**——ROI 说"该投多少时间"，自评说"当前我站在哪里"。两者结合才能做对决策。

## L6：元认知层级——知道自己知道什么，知道自己不知道什么

### 定义

> **L6** = 在每个方面**知道自己知道什么 & 不知道什么**。
> - "知道" = 真的实践过、学过、读过、摸过、用过
> - "不知道" = 确实了解过（有 awareness），但**没时间深入过，更没摸过用过**
> - **"相对"** = 即使到了 L6，也要承认**一定还有边界是自己不知道的**（未知的未知永远存在）

### 为什么 AI 时代特别需要 L6

AI 协作有个**结构性副作用**：它把原本该逼你遇到的"暗礁"平滑掉了。

- 过去部署一套 Docker + Caddy + Python 多服务，你会被迫读 10 篇文档、踩 5 个坑、搜 20 个 StackOverflow。每个坑都暴露一个"我不知道的领域"——**未知被动扫描出来**
- 现在 Claude 一次对话生成完整配置，你复制能跑。**所有暗礁都被 AI 平滑掉了** → **未知没被扫描 → 你不知道自己不知道**

**L1 自评陷阱**说"承认 AI 辅助跑通只是 L1"。
**L6** 说：**光承认单点 L1 不够，还要主动扫描 + 诚实标记整个地图**。

### L6 与 L1-L5 的区别

| 维度 | L1-L5 | L6 |
|------|-------|-----|
| 对象 | 单个技术点 | 跨技术的地图 |
| 方向 | 纵向深度 | 横向广度 + 元认知 |
| 测量 | "我在 X 上多深" | "我对 X 领域的认知有多清晰" |
| 外部价值 | 技术深度可验证 | **资深工程判断力可验证** |
| 独立性 | 可以只有 L5 没 L6（单点深但地图糊） | 可以 L6 不 L5（地图清但没深点） |

**真正的 senior = L4-L5 的几个深点 + L6 的清晰地图**。

### L6 的三层结构

用"我对 LLM 可观测性领域的认知"举例：

**第一层 · 已知（真实做过）**
- Langfuse 自托管 v3，运维 4 月（L4-L5）
- OTel Collector 配置，读 source，写 processor（L4）
- LangChain callback 集成 Langfuse（L3-L4）

**第二层 · 已知未知（了解过但没实操）**
- LangSmith：读过文档，知道它是 Langfuse 的商业对手，没对比过 trace schema 差异
- Helicone：听过名字，知道是 LLM proxy + observability，没部过
- Arize Phoenix：读过一篇 blog，知道它偏 eval 方向，没实操
- OpenLLMetry：听过，是 Traceloop 做的 semantic convention，没深入

**第三层 · 承认未知的未知**
- 这个领域还在快速演化，2024-2026 之间可能有我完全没听过的新玩家
- 大厂内部工具（Anthropic / OpenAI 内部 trace 系统）我不可能知道
- 某些细分场景（比如 RAG 专属 observability）可能有我没关注的专门方案

**这三层合起来 = L6 在"LLM 可观测性"这一个领域的体现**。

### 用词纪律（L6 的核心外显特征）

L6 的人说话有**精准的用词分级**：

| 不准确 | 精准 |
|--------|------|
| "我熟悉 X" | "我在 X 上运维了 4 月，处理过 Y 次事故" |
| "我了解 Y" | "我读过 Y 的文档和一篇 blog，没实操过" |
| "我知道 Z" | "我听过 Z 的名字，只知道大概是干什么的" |
| "我会这个领域" | "这个领域里我做过 A，读过 B，听过 C，可能还有 D 我不知道" |

**模糊词（熟悉/了解/知道/会用）= 认知懒惰**。
**精准动词（摸过/读过/听过/没接触）= L6 诚实**。

### 怎么达到 L6

L6 不能靠学 —— 要**主动做认知地图**。

**动作清单**：

1. **每个深投入的技术领域建一张地图**（markdown 文档，见 [`knowledge-maps/README.md`](./knowledge-maps/README.md)）
   - 列三层：已知（做过）/ 已知未知（听读过）/ 未知的未知（承认）
   - 触发式更新：做了事 / 读了文档 / 听到新名字 → 当时就改，不过夜

2. **定期做 landscape scan**（每月 1 小时，固定日期）
   - 随便打开一个领域（比如 "LLM Gateway"），搜 "top 10 alternatives 2026"、"XX vs YY"
   - 不是为了学完，**是为了把"听过的名字"扩充**
   - 把每个找到的新名字分类到三层

3. **写作/讲述暴露未知**
   - 试图向别人解释时卡壳的地方 = 未知
   - 对外讨论追问答不出的地方 = 未知
   - 暴露出来后标进地图第二层

4. **决策时主动留弹性**
   - 架构选型留 "未来发现更好方案可替换" 的抽象层
   - 时间估算留 buffer 给"可能遇到我没预料的坑"
   - 这不是保守，是对"未知的未知"的理性应对

**具体操作规则**：见 [`knowledge-maps/README.md`](./knowledge-maps/README.md)——8 条维护规则 + 模板 + 触发器 + 诚实校准自检。

### L6 的外部价值

**初级读者** get 不到 L6 的价值（他们自己还在 L3-L4），不会深究。

**资深工程师 / 技术决策者**会立刻看出区别：

- 读者读同行的文字，最怕"瞎自信的 L4"——技术深但看不清边界，上来就拍板
- L6 的人"我做过 X，Y 我没做过但知道大概，Z 我可能不知道"——**不会瞎拍板**
- 这种人的文章 / 方案 / 代码更值得信任

**L6 表达 vs 伪谦虚**：

- ❌ 伪谦虚："我也就一般般懂"（实际啥都没想清楚）
- ✅ L6："A 我运维过 4 月，B 我读过没实操，C 我听过名字，这领域还有我不知道的玩家。做选型我会先花 3 天做 landscape 扫描再决定"

**L6 = 具体 + 诚实 + 有后续行动**，不是空话。

### 警告：别把 L6 当装谦虚

L6 的核心是**诚实 + 地图**，不是**"对什么都说我不懂"**。

反例：
- 明明在某领域 L4-L5，但出于"谦虚"说成 L2 → 不是 L6，是虚假 humility
- 明明某领域全是 L0，装作 L6 说"我承认有未知" → 不是 L6，是包装

**L6 的前提是你在某些领域有真实 L4-L5 深度**。没有 L4-L5 支撑的"承认未知"只是无知。

公式：**真正的 L6 = 有点深 × 地图清晰 × 承认边界**。三者缺一不可。

## 三个具体约束推论

### 推论 1：不要在 AI 可替代的事上花大时间

**反例**：

- 花 3 天研究"Docker 多阶段构建最佳实践"——AI 10 分钟能给你 90 分答案
- 花 1 周读 Langfuse 本地源码——AI 协作下 2 天能走完主链路

**正例**：

- 花 2 周**线上运维** Langfuse，遇到升级不兼容 / ClickHouse merge 失败 / 备份恢复 / 证书续期挂 → **每个坑 AI 帮不了你**，只能你现场判断

### 推论 2：真实运行时间是最稀缺资源

**核心事实**：

- "我跑了 4 个月" 这句话本身就有价值
- 因为 4 个月你没法压缩，读者无法怀疑你造假
- 4 个月里遇到的真实事故（磁盘爆、OOM、证书过期、DNS 劫持等）是**独家素材**

→ **越早开始线上运行越好**。4 月开始部署，到 8 月刚好 4 个月真实运行。

### 推论 3：故障经验 > 搭建经验

**同行读者视角对比**：

❌ "我搭过 Langfuse"
→ 读者：AI 都能搭，所以呢？

✅ "我运维 Langfuse 4 个月，遇到过一次 v3.38 升级后 ClickHouse schema 迁移失败，追到 upstream issue #5234，临时回滚方案是 X，永久 fix 是 Y"
→ 读者：**这个值得继续读 / 值得深聊**。

差别是什么？

- **搭建**：一次性动作，AI 完美辅助
- **故障**：时间维度的经验，AI 无法帮你"经历"

→ 推论：**主动制造故障演练**，让 4 个月密度最大化。

## 反方案的错误演化（作为反例）

本专题的方案经历了 6 轮推理才收敛（含视角校准）。每一轮都犯了同类错误：

### R1: "用 SaaS 省心"

- 错在：**把"运维便利"当目标**，但这个目标对外部读者 / 评估者都不关心
- 学到：别在工程师作品集方案里优化"省心"

### R2: "全自托管为故事"

- 方向对，理由弱——没说清为什么故事有价值
- 学到：故事价值要落到"AI 能不能替代"这个根本维度

### R3: "资源 ROI 算账"

- 错在：**把 ROI 当成资源利用率**，算￥200 够不够
- 用户反驳："学习投入产出比，不是资源问题"
- 学到：ROI 有学习、资源、时间、分享价值四维，不同维度结论差 10 倍

### R4: "学习 ROI 分级"

- 错在：**学习深度 ≠ 分享价值**
  - 本地学透 Langfuse 源码 = 学习深度高，但 AI 替代难度低 → 分享价值低
  - 线上运维 Langfuse = 学习深度中，但 AI 替代难度高 → 分享价值高
- 学到：要叠加"AI 替代难度"这个维度才是真 ROI

### R5（收敛）: "线上真实运维 4 个月 = 护城河"

- 所有约束都能统一解释
- 成本问题、技术选型、时间分配全部导出一致结论

### R6（视角校准）: "主体是技术分享者，不是求职者"

- 错在：**前 5 轮默认叙事主体是'求职候选人'**，导致文档大量"面试官"、"offer"、"简历"语境
- 用户校准："这是我做技术分享的视角"
- 学到：求职是副产出；主产出是持续输出 + 建立专业信用。分享者视角下所有结论依然成立，但**叙事更长期、更复利**

## 这个原则的其他应用

不只用于部署方案，整个 4-6 个月技术栈规划都适用：

| 决策场景                   | 用原则怎么判断                                                                             |
| -------------------------- | ------------------------------------------------------------------------------------------ |
| "要不要学 K8s？"           | 学完能不能拿到 L4-L5 经验？（能，如果真跑一个集群 4 月；不能，如果只是本地 minikube 玩玩） |
| "要不要接 RAG？"           | 线上用户真用的 RAG 才是 L4；本地 demo RAG 是 L2                                            |
| "要不要做 eval pipeline？" | 做了且真跑 100 次 eval 积累 = L4；demo eval = L2                                           |
| "要不要写 blog？"          | 基于真实经验写 = 有价值；基于文档翻译写 = 无价值                                           |

**简单自测题**：

> "如果有读者或同行追问这段经验，我的回答能不能让他深挖 15 分钟而不穿帮？"

如果不能，说明这段经验还在 L1-L3，**需要加大真实运行投入**。

---

## 警告：不要过度演绎这个原则

**原则 ≠ 教条**。有几个不适用的场景：

1. **基础技能不补**：Docker / Git / Linux 基本功即使 AI 能替代也要自己会，不然 L4-L5 根本撑不起
2. **过度追求稀有**：不要为了"AI 替代不了"去学冷门技术，没人讨论也没人关心就白搭
3. **忽视交付价值**：线上运行的前提是先"线上能跑"，别为了积累运维故事把 MVP 无限推迟

正确姿势：**先把 MVP 上线 → 然后最大化真实运行时间 → 沿途主动制造故障**。

## 给自己的承诺

作为原则的具体行为约束：

- [ ] 每个组件部署完就**上线跑**，不留本地
- [ ] 每周记录至少 1 条线上观察（正常 / 异常都行）
- [ ] 每月主动演练 1 次故障
- [ ] 遇到真实事故必写**事后复盘**（`07-incident-log.md`），同时作为 blog/video 草稿原料
- [ ] 做任何对外能力陈述前，每条经验都用 L1-L6 自评，**L3 以下不公开说**
- [ ] 任何"AI 协作跑通过"的经历，自评**默认 L1**，除非能列出独立应对的具体问题
- [ ] 不用"已会"、"已掌握"、"熟练"等模糊词自评——用具体事实（运维 X 月 / 事故 Y 次 / commit Z 条）代替
- [ ] **为每个深投入的技术领域建一张 L6 地图**（已知 / 已知未知 / 承认未知），每季度更新
- [ ] 用词精准到动作：**"摸过 / 读过 / 听过 / 没接触"**，不用"熟悉/了解"
- [ ] 做技术选型时**主动留弹性**——给"我可能不知道的方案"留抽象层
