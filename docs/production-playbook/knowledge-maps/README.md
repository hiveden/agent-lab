# Knowledge Maps — L6 元认知地图维护规则

> **定位**：这个目录存放**领域级认知地图**，不是技术笔记、不是教程、不是学习资料。
> 每张地图只记**状态标签**（做过/读过/听过/没接触），不记知识本身。
> 本文档**只写规则**，具体地图在后续工作中自然填充。

---

## 核心规则（一页纸能看完）

### 规则 1：用词纪律（强制）

**只能用这 4 个精准动词**：

| 动词 | 含义 |
|------|------|
| **做过** | 真实实操过——部署过 / 修过 bug / 读过源码到调用链 / 线上运维过 |
| **读过** | 只读过文档/blog/论文，没实操过 |
| **听过** | 只知道名字和大概定位，没读过资料 |
| **没接触** | 完全不知道存在（隐含——承认第三层） |

**禁止用**：熟悉 / 了解 / 知道 / 会用 / 掌握 / 精通

违反即重写。

---

### 规则 2：领域清单（先列清楚再填）

核心领域（建议 4-6 个，agent-lab 相关）：

- `llm-observability.md` — Langfuse / LangSmith / Phoenix / Helicone / OpenLLMetry 等
- `llm-gateway.md` — LiteLLM / Portkey / OpenRouter / Kong AI Gateway 等
- `agent-architecture.md` — LangGraph / AutoGen / CrewAI / LlamaIndex Agents 等
- `deployment-runtime.md` — Docker / K8s / Fly.io / Cloudflare / PaaS 选型
- `observability-general.md` — OTel / Grafana / SigNoz / Datadog / Prometheus 等
- `rag-memory.md` — 向量库 / RAG pattern / memory 方案（如涉及）

**规则**：领域要**稳定**，不要随便加减。遇到新技术**归类到现有领域**，不为一个新名字开新领域。

---

### 规则 3：地图文件结构（统一模板）

每张地图严格按此结构：

```markdown
# [领域名]

**最后更新**: YYYY-MM-DD
**整体状态**: 主深点 LX（引用 L1-L5）+ L6 地图 [粗/中/清晰]

---

## 第一层 · 做过（L3+ 深度）

- **[技术名]**：[1-2 行具体事实] | [证据链接/commit]
  - 例：`Langfuse v3 自托管生产运维 4 个月，2 万条 trace，处理过 N 次事故 | GitHub issue #X`

## 第二层 · 读过/听过

- **[技术名]**：[读过文档？读过 blog？对比过？仅听名字？] | [接触时间]
  - 例：`LangSmith：读过官方文档首页 + API reference，未实操 | 2026-04`
  - 例：`Helicone：只听过名字，知道是 LLM proxy + observability`

## 第三层 · 承认的盲区（命名层）

- [这领域可能还有我不知道的类型 X]
- [某个细分场景（如 RAG observability 专属方案）我没关注]
- [某类玩家（如大厂内部工具 / 新兴 2025-2026 产品）我不可能接触]

## 选型方法论（面试话术原料）

- 如果要做选型，我会：[X 天 landscape scan] → [选 top 3 做 POC] → [按 Y 标准打分]
- 我的**默认选择**和**理由**：...
- 我**会警惕的选择**和**理由**：...
```

**关键**：不写技术本身怎么用、怎么配置、怎么排坑——**那些放 blog / incident-log / runbook**，不进地图。
地图只回答：**"我在这个领域的认知覆盖有多广，每块多深？"**

---

### 规则 4：更新触发器（什么时候更新，不等于"有空更新"）

**触发式（事发当时就改，不过夜）**：

| 触发事件 | 动作 |
|---------|------|
| 完成一次真实实操（部署成功 / 修了 bug / 读完源码一段） | 对应技术从"读过"→"做过"（或"听过"→"读过"） |
| 读了一篇某技术的 blog / 文档章节 | "听过"→"读过"，标注读的具体是什么 |
| 看到一个陌生技术名（HN / Twitter / 同行讨论） | 加到"听过"层 |
| 事故 / 调试时发现"这东西我原来不知道" | 从第三层（承认未知）挪到第二层（命名盲区） |
| 面试模拟被卡住 | 标记卡点 → 归类到对应地图的第二/三层 |

**定期式**：

| 频率 | 动作 |
|------|------|
| **每周 review（15 min）** | 扫一遍本周 commit / blog draft / 事故记录，看哪些触发了更新但忘了改 |
| **每月 landscape scan（1 小时）** | 一个领域搜一次 "top alternatives 2026" / "XX vs YY"，补"听过"层 |
| **每季度大 review（2 小时）** | 所有地图整体复审：删过时的、提升已升级的、重审第三层的盲区假设 |

---

### 规则 5：不做什么（防止地图变成博客）

- ❌ **不写使用教程**（那是 blog 的活）
- ❌ **不写最佳实践**（那是 runbook 的活）
- ❌ **不写"我为什么选 X"的长论述**（那是事故复盘或 ADR 的活）
- ❌ **不为凑数添加"读过"条目**（宁缺毋滥，假数据毁掉面试可信度）
- ❌ **不把"看过标题"算"听过"**（听过要能说出大概定位）
- ❌ **不把"跟 AI 聊过一次"算"读过"**（必须独立读过文档/blog 的具体章节）

---

### 规则 6：诚实校准（防止自我欺骗）

每次更新前过一遍自检：

- **做过自检**：能讲 10 分钟不穿帮吗？有 commit / 日志 / 数字吗？ → 不能就降级为"读过"
- **读过自检**：能指出具体读的哪一部分？有笔记或 draft 吗？ → 说不清就降级为"听过"
- **听过自检**：能说出它的核心定位和一个区分特征吗？ → 说不出就删掉这条
- **盲区自检**：我写的第三层是真承认，还是凑数？真承认的标志是"我能想象它可能是什么，但我不确定" → 凑数就删

**关键纪律**：**宁可地图看起来少，不要地图看起来多**。空地图 + 真实 > 膨胀地图 + 伪造。

---

### 规则 7：和对外表达的连接

- 对外表达前（blog 发布、技术分享、简历更新、面试等）：每张地图导出成速查卡
- 被问"你怎么看 XX 领域" → 30 秒内按三层说完
- 地图第二层 + 第三层 = **L6 陈述的主战场**（证明认知地图清晰 + 诚实）
- 简历/作品页的"技术栈"按本目录地图结构写（见 [`04-expression-precision.md`](../04-expression-precision.md) "L6 版的简历表达"）
- 地图本身也是 blog 选题库：第二层"读过没实操"→ "我如何从 X 入门"系列；第三层"承认未知"→ "XX 领域我还不知道什么"系列

---

### 规则 8：优先级（不强求一次建全）

**不要一次建 6 张**。优先级：

1. **先建你当前最深入的 1 张**（比如 4-6 月运维 Langfuse 后，先建 `llm-observability.md`）
2. **3 个月内建齐 3 张**核心领域
3. **内容收口期（2026-07）前建齐 4-6 张**
4. 不相关的领域不建（不为"看起来全面"硬凑）

每张地图建起来后，靠**触发式更新**维持，不需要重建。

## 执行承诺

这份规则落到行为上，必须在 agent-lab 日常工作中自动触发：

- [ ] **写 commit 时自问**：这个 commit 有没有把某技术从第二层升到第一层？有就改地图
- [ ] **读 blog/文档时自问**：这个内容能让某条从"听过"升到"读过"吗？能就改
- [ ] **遇到陌生名字时**：立刻加到对应领域第二层，不要假装没看见
- [ ] **事故复盘写完**：回头看这次事故暴露了什么盲区，更新第三层
- [ ] **月初 1 小时**：做 landscape scan（预先固定日期，比如每月 1 号上午）

**这 5 个动作不做 = 规则空转 = L6 停留在纸面**。
