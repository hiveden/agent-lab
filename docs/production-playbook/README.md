# Production Playbook — 生产化 × 技术分享资产积累

> **定位**：这不是技术文档，是**战略文档**。
> 用 agent-lab 作为载体，在 2026-04 → 2026-08 的 4-6 个月窗口内，建立"AI 时代 AI Agent 工程师"的**可持续技术分享体系 + 不可替代的真实运维经验**。

---

## 专题背景

**时间节点**：2026-04 → 2026-08 的 4 个月密集产出期
**投入**：4 个月脱产 + 学费，综合成本 10 万+
**主目标（按优先级）**：

1. **建立可持续的技术分享体系**——blog / video / demo，基于真实运维经验
2. **积累 L4-L5 级别的真实经验**——不可替代的运维时间和故障履历（分享的原材料）
3. **构建 L6 认知地图**——跨领域的专业判断力
4. **次要产出**：外部机会评估（面试 / 接项目）——主目标做到位这些是自然结果

**核心约束**：
- 不能把时间花在 AI 可替代的"本地搭建"类经验上
- 不能纠结 ￥200-500/月 的运维小成本（相对 10 万是噪音）
- 不能做"学了但讲不出故事"的事

---

## 核心原则（一句话总结）

> **AI 协作时代，工程师的护城河 = 决策链路长 + 不可重放 + 真实时间积累的经验。**

展开：
- AI 1 小时能做的事 → 没有分享价值（读者能让 AI 自己生成）
- AI 能带你做但需要你判断的 → 中等价值，不够差异化
- **AI 做不了，只有"真跑过 N 个月 + 真踩过坑"的经验 → 真护城河**

→ 详见 [`01-principles.md`](./01-principles.md)

## 学习 ROI 分级框架

按"分享价值 × 通用杠杆 × 行业主流度"给每个技术打分：

| 级别 | 代表技术 | 投入建议 |
|------|---------|---------|
| **S** | OpenTelemetry / Langfuse（生产运维） | 1-2 周深度投入，必须讲到原理 |
| **A** | LiteLLM Gateway / Docker 多服务 / Grafana 生态 | 2-3 天专项 + 运维持续积累 |
| **B** | Cloudflare Pages+D1 / Fly.io | **运维载体型**——不单独投入深度时间，靠真实部署 4-6 月自然积累（现状 L1，目标 L3） |
| **C** | SigNoz / GlitchTip 深度自托管 | 跳过深度，但**线上跑**积累运维经验 |
| **D** | Sentry Cloud / SaaS 配置 | 0 深度，会用就行 |

→ 详见 [`02-learning-roi.md`](./02-learning-roi.md)

## 部署方案（结论）

**两节点 + Cloudflare 边缘 + 自托管为主 + 少量 Cloud 兜底**：

```
Cloudflare Pages (BFF + D1)              ← 边缘部署
         ↓
┌─── 阿里云 2GB (hkg-1) 应用节点 ───┐
│  Python Agent (Radar)              │
│  LiteLLM Proxy                     │
│  Caddy (入口)                       │
└────────────────────────────────────┘
         ↓ Tailscale 内网
┌─── Hetzner 8GB (eu-1) 观测栈节点 ──┐
│  Langfuse v3 自托管                │
│  SigNoz 自托管                     │
│  GlitchTip 自托管                  │
│  OTel Collector                    │
│  Caddy (子域名反代)                 │
└────────────────────────────────────┘
```

**关键决策**：
1. 全组件**线上跑 4-6 个月**——不纠结"资源足不足"，纠结"运维经验够不够"
2. **跨云跨 region**——为了获得跨境运维经验（证书、DNS、Tailscale、CDN）
3. **故意制造故障演练**——没遇到就创造，积累排障履历（也是内容素材）

→ 详见 [`03-deployment-plan.md`](./03-deployment-plan.md)

## 时间分配：20% 部署 + 80% 运维 + 持续输出

| Phase | 周 | 重点 |
|-------|----|------|
| **1. 部署** | W1-2 | 全栈上线 + 首篇 blog 产出 |
| **2. 告警** | W3 | Prometheus alert + Telegram bot |
| **3. 故障演练** | W4 | 主动杀进程、填磁盘、断网（每次演练 = 1 篇 blog 素材） |
| **4. 调优** | W5-6 | TTL、缓存、备份自动化 |
| **5. 真实运行积累** | W7-16 | 就让它跑，记录每次事故（blog/video 原材料） |
| **6. 内容收口 + 公开发布** | 面试期前 | blog 精修、demo 视频、GitHub 作品页 |

**Phase 5 是内容原材料的黄金期**——真跑才会有真事故，真事故才能写出 AI 不能代笔的内容。这也是深度追问时候（同行讨论 / 面试深挖）唯一靠得住的素材来源。

→ 详见 [`03-deployment-plan.md`](./03-deployment-plan.md)

## 技术陈述精度分级

| 层级 | 陈述形式 | AI 替代难度 |
|------|---------|-----------|
| L1 | "我用 SaaS" | AI 完全能做 |
| L2 | "我本地自托管了一套" | AI 能带你做 |
| L3 | "我本地 + 生产，做了 ROI 权衡" | AI 能推 |
| **L4** | **"我生产运维 4 个月，踩过 X 个坑"** | **AI 替代不了** |
| **L5** | **"踩坑细节 + 监控指标 + 修复 commit + 事后复盘"** | **完全独家** |
| **L6** | **"A 我做过，B 我读过没实操，C 我听过，还有 D 我不知道；做选型会先 landscape 扫描"**（元认知 + 地图诚实 + 承认未知） | **资深工程师级稀缺项** |

**目标**：**L4-L5 的几个深点 + L6 的清晰地图**。
**路径**：深点靠"真跑 4 个月"熬；地图靠每月主动 landscape scan + 诚实用词。

→ 详见 [`04-expression-precision.md`](./04-expression-precision.md)

## 决策演化记录（方法论 meta 材料）

本专题的方案经历了 5 轮推理才收敛，每一轮都被用户原则性反驳修正。演化轨迹本身是宝贵素材——**方案不是一次想对的，是在真实约束下反复校准的**。

| 轮次 | Claude 方案 | 用户反驳 | 学到的 |
|------|-----------|---------|--------|
| R1 | SaaS 省心 | 分享价值归零 | 省心 ≠ 省力 |
| R2 | 全自托管，故事导向 | 资源 ROI 没算 | 故事要落到资源约束 |
| R3 | 资源 ROI 分析 → 混合方案 | 我问的是学习 ROI | ROI 有多种定义 |
| R4 | 学习 ROI 分级 → 部分自托管 | 忽略 AI 替代性维度 | 学习深度 ≠ 分享价值 |
| **R5**（收敛） | **线上全栈 + 运维时间 = 护城河** | ✓ | 最关键的是**不可替代性** |
| **R6**（视角校准） | **主体是求职者** | 应该是技术分享者 | 分享者视角是更长期的复利叙事 |

这段演化作为 appendix 保留——它本身就是一篇 blog 选题：**"我如何识别自己思路的漏洞并校准"**，展示的元能力比任何具体技术结论都更稀缺。
