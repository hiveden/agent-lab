# 02. 学习 ROI 分级框架

> **用途**：决定在每个技术上投入多少时间、是否自托管、是否深度钻研源码。

---

## 评分三维度

每个技术按三个维度打分：

| 维度 | 含义 | 高分特征 |
|------|------|---------|
| **面试频率** | 小公司 AI Agent 岗被问到的概率 | AI Agent 标配、后端通用基础 |
| **通用杠杆** | 脱离当前项目这个技能还值不值钱 | 非 AI 领域也用 / CNCF 事实标准 |
| **行业主流度** | 是不是目前生产的首选 | 大厂都用 > 小众但开源 > 一家独大的闭源 |

**综合评级**：S / A / B / C / D

---

## 各组件评分表

### S 级（必须深度投入，1-2 周）

| 技术 | 面试频率 | 通用杠杆 | 主流度 | 备注 |
|------|---------|---------|--------|------|
| **OpenTelemetry** | 🔴🔴🔴 | **极高**（非 AI 也用） | CNCF 毕业 | 最大杠杆，离开 AI 也是硬通货 |
| **Langfuse（运维）** | 🔴🔴🔴 | 中（AI 限定） | LLM trace top3 | AI Agent 岗面试核心话题 |

**投入建议**：
- OTel：学 SDK + Collector + 采样策略 + W3C trace-context，1-2 周
- Langfuse：**线上跑 4 个月**为主，本地读源码 3-5 天

### A 级（中度投入，2-3 天专项 + 运维持续积累）

| 技术 | 面试频率 | 通用杠杆 | 主流度 | 现状自评 | 目标 |
|------|---------|---------|--------|---------|------|
| **LiteLLM Proxy** | 🟡🟡 | 中 | 开源 LLM Gateway top1 | L0-L1 | L3-L4 |
| **Docker 多服务 + 反代** | 🔴🔴🔴 | 高 | 后端基本功 | **L1**（AI 辅助跑通过 tts 项目，没独立遇过复杂问题） | L3-L4 |
| **Grafana 生态（LGTM）** | 🔴🔴 | 高 | 行业主流 | L0 | L3 |

**投入建议**：
- LiteLLM：配 fallback / cache / cost tracking / routing，2-3 天专项 + 生产运维持续积累
- Docker/Caddy：**不当"已会"**——双节点 + 反代 + HTTPS + 跨云网络会带来 tts 没遇到过的问题，预留 1 周专项学习 + 持续踩坑
- Grafana：注册 Cloud + LGTM 栈接入，2-3 天专项

### B 级（运维载体型——不单独深度投入时间，依托生产运维自然积累）

**定义**：这些技术本身不作为"专项学习目标"，但**会作为 agent-lab 部署链路的载体持续使用 4-6 个月**。通过真实运维积累经验，不通过"专门学一下"。

| 技术 | 面试频率 | 通用杠杆 | 主流度 | 现状自评 | 目标 |
|------|---------|---------|--------|---------|------|
| **Cloudflare Pages + D1** | 🟡 | 中 | Edge 热但 D1 小众 | **L1**（AI 协作跑通过本地 dev，没生产运维） | L3 |
| **Fly.io 部署** | 🔴🔴 | 中 | 独立开发者主流 | **L1**（跟 AI 跑通过 tts-agent-harness，没独立排障经验） | L3 |

**关键认知**：
- **不要把"AI 辅助跑通过"当作"已掌握"**——这只是 L1
- **也不要为它们专门安排深度学习周**——投入产出比不如 S/A 级
- **正确做法**：让它们作为真实部署链路的一部分跑起来，遇到问题就解决，在 4-6 月运维过程中自然积累
- **可对外陈述的素材**：4-6 月后真实运维事故 / 数据 / commit，不是"我搭过"

**自评 L1 的具体含义**：
- 能跟 AI 一起把流程走通，但**独立遇到非文档化问题时判断力不足**
- 没碰过 Cloudflare Workers 的 CPU limit、D1 的并发限制、Fly.io 的 OOM 行为
- 没碰过部署被证书过期 / DNS 劫持 / 供应商 downtime 拖停的真实场景
- 这些都要靠 Phase 5 真实运行才能积累

### C 级（深度学习跳过，但**线上跑积累运维经验**）

| 技术 | 面试频率 | 通用杠杆 | 主流度 | 为什么不深度学 |
|------|---------|---------|--------|-------------|
| **SigNoz** | 🟢 | 低 | 小众（Grafana 是主流） | 深度学 ClickHouse 调优价值低 |
| **GlitchTip** | 🟢 | 低 | 很小众 | 错误监控不是 AI 热点 |

**投入建议**：
- **不读源码、不调优、不研究内部架构**
- **但要线上跑**，积累"运维过一个 Django + Celery 服务 / ClickHouse 数据库"的经验
- 线上运维经验可以讲：**"我运维过一个 ClickHouse 生产实例"** → 这个经验**是 AI 替代不了的**，即使技术本身不主流

**关键 insight**：SigNoz 深度学习 ROI 低，但**线上运维 SigNoz 的 ROI 高**。这两件事分开算。

### D 级（零深度，用就行）

| 技术 | 建议 |
|------|------|
| **Sentry Cloud** | 注册账号，配 DSN，完事 |
| **Langfuse Cloud**（如果选这条） | 生产兜底用，不是学习目标 |

## 总时间分配

按 ROI 级别分配 4-6 个月的深度学习时间：

```
S 级：   40% （OTel 1-2 周 + Langfuse 源码 3-5 天 + 运维 4 月 background）
A 级：   30% （LiteLLM + Docker + Grafana，专项 + 运维持续）
B 级：   10% （运维载体型，无专项学习时间，靠真实部署触发学习）
C 级：   15% （线上运维投入，无深度学习）
D 级：   5%  （SaaS 配置）
```

**time budget**：约 **3-4 周专项深度学习** + **4-5 个月持续线上运维**。

**重要**：B 级的 10% 不是"复盘"——是"作为部署链路跑起来后，遇到 Cloudflare / Fly.io 真实问题时的学习时间"。现状自评 L1，靠持续运维升到 L3。

## 动态调整规则

ROI 评级不是死的，4-6 个月中可能调整：

**升级触发**：
- 面试反馈某技术被频繁问到 → 升级 B→A 或 A→S
- 发现某组件踩坑价值超预期 → 升级 C→B

**降级触发**：
- 学完发现面试没人问 → 降级
- 行业趋势变化（比如新 LLM Gateway 出来) → 降级 LiteLLM

**评估节奏**：
- 每月 review 一次，在 `07-incident-log.md` 里记录

## 关键争议点

### 为什么 SigNoz 不深度学但要线上跑？

看起来矛盾，其实不是：
- **深度学习 SigNoz 内部**（ClickHouse 调优源码） = 小众技术，面试问不到 = 低 ROI
- **线上运维 SigNoz** = 运维过一个大型开源 Django + Celery + ClickHouse 栈的经验 = **跨项目通用**

所以：**用 SigNoz 做"运维经验载体"**，不是"SigNoz 知识本身"。

类比：你开卡车学了驾驶技能，不是学了"卡车本身"。驾驶技能跨车型通用。

### 为什么 Grafana Cloud 在 A 级，SigNoz 只在 C 级？

同样是 trace/metric/log 方向，但：
- Grafana：**行业主流**，大厂几乎都在用（LGTM 栈 = Loki/Grafana/Tempo/Mimir）
- SigNoz：**小众**开源替代方案，面试很少问

学习 ROI 看**行业需求**，不是看**产品好坏**。

### 为什么不学 K8s？

当前规划没包含 K8s，因为：
- K8s 学习曲线陡（1-2 月起）
- agent-lab 双节点用 Docker + Tailscale 完全够
- 专门学 K8s 建议另开一个项目（比如把 agent-lab 迁移到 K8s 作为后续作品）

**但如果 4-6 月中确实有时间**：建议最后 1 个月把 agent-lab 迁移到单节点 K3s，作为**额外加分项**——不必要但锦上添花。

## 使用本框架的自查清单

每次做技术选型前自问：

- [ ] 这个技术我要投入几天？
- [ ] 几天后能拿到 L3 还是 L4 经验？
- [ ] 行业主流度如何（不是产品好坏）？
- [ ] 面试问到这个的频率？
- [ ] 如果不深度学，有没有替代方式（比如"只运维不读源码"）？

答完这几个问题，ROI 级别基本出来了。
