## 今天的失误与重要纠偏

### 1. UI 原型走过弯路(三版)
- **v0.1**:Generic SaaS dark mode,被用户 "UI 太丑"
- **v0.2**:用 frontend-design skill 做了编辑式戏剧海报(Instrument Serif + 罗马数字),被用户 "戏剧海报？我们项目是什么？" 打醒 — 形式压过内容,不适合工程师工具
- **v0.3**:克制的 Linear/LangSmith 风(现采用),获用户 "高级"

**教训**:工具的设计语言必须匹配用户的心智模型。工程师调试 Agent 不是欣赏排版。

### 2. "先拆成两个视图"的判断是错的
被用户反问 "看看 Linear、Height、Raycast、Arc",意识到这些产品证明了**单视图完全能装下 list+detail+inspector**,关键是 sidebar-nav + detail-as-main + inspector-on-demand + cmd+k + 键盘层。
**教训**:不要因为自己不会拼就说"不能拼"。

### 3. 技术闭环 ≠ 产品闭环(今日最大反思)
一整天都在做"数据能流起来",但没回答"用户要做什么"。用户在最后问:
- 信息源如何配置?
- 如何查看和修改配置?
- 如何修改提示词?
- 如何管理资讯?

**4 个问题,我 4 个都没设计**。

Radar 真正的最小产品闭环应该是:
```
发现 → 阅读 → 判断 → 反馈 → 学习 ──┐
  ↑                                  │
  └──────────────────────────────────┘
```
现在**只做了前两个**,后三个完全空缺。这不是一个能自运转的产品,只是一个可以手动触发的脚本 + 展示页。

**教训**:技术 ready 不是产品 ready。先定产品动作链条,再想怎么技术实现。

## 明天待讨论的产品设计问题

### 最小产品闭环应包含(待用户确认)
1. **自动定时采集** — 不是手动 trigger,是"你不管它它自己推"
2. **Watching 队列** — 标 watching 后有地方回顾,不然没意义
3. **反馈字段** — 点 dismissed 时弹一行 note,存 D1
4. **反馈 → prompt 注入** — 下次 push 时读最近反馈作为 learned_rules 注入 Agent system prompt
5. **Runs 历史表** — 可追溯每次 push 的时间 / 结果 / 错误

### 二级 tab 设计(不是 MVP,是中期方向)
- `Feed`(现有)
- `Runs` — push 历史
- `Sources` — 信息源管理
- `Prompts` — 提示词编辑器(有版本)
- `Settings` — LLM / 调度 / 阈值

### 数据模型演进(如果走产品化)
```sql
runs            (id, agent_id, trigger, started_at, finished_at, status, inserted, skipped, spans_json, error)
sources         (id, agent_id, kind, config_json, enabled, last_fetch_at)
prompts         (id, agent_id, key, content, version, created_at)
agent_settings  (agent_id, key, value_json)
-- items 新增: run_id / feedback_note / archived_at
```
